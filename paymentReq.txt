# 🧱 BACKEND REQUIREMENTS

## 1. Ledger-Based System

Create a `rent_ledger` table:

* tenant_id
* pg_id
* month (DATE, first day of month)
* rent_amount
* paid_amount (default 0)
* balance_due
* prev_balance (carry-forward)
* status ('due', 'partial', 'paid')
* due_date (e.g., 5th of every month)
* created_at

Constraints:

* UNIQUE (tenant_id, month)

---

## 2. Monthly Auto Due Creation

Implement a CRON job:

* Runs on 1st of every month
* Creates a new ledger entry for all active tenants
* Sets:

  * rent_amount = monthly_rent
  * paid_amount = 0
  * balance_due = monthly_rent
  * status = 'due'

Use:

* ON CONFLICT DO NOTHING

---

## 3. Payment Logic (CRITICAL)

When a payment is made:

### Step 1: Allocate to previous dues first

* If prev_balance > 0 → deduct from it first

### Step 2: Remaining goes to current month

### Example:

* Previous due = 4000
* Current rent = 6000
* Payment = 5000

Result:

* Previous cleared = 4000
* Current paid = 1000
* Current due = 5000

---

## 4. Update Ledger Properly

After every payment:

* Update:

  * paid_amount
  * balance_due
  * prev_balance
  * status

Never overwrite blindly — always calculate from existing data.

---

## 5. Prevent Duplicate Payments

* Do not allow payment if:

  * tenant has no outstanding dues
* Dropdown should only include tenants with:

  * balance_due > 0 OR prev_balance > 0

---

## 6. Delete Payment Logic

* When payment is deleted:

  * Recalculate ledger
  * Restore previous + current balances correctly

---

## 7. Tenant Status Handling

* If tenant is vacated:

  * Do NOT delete payment history
  * Mark status = 'vacated'
  * Exclude from active UI

---

## 8. Email Reminder System

* Run daily cron job
* If:

  * due_date < today
  * AND balance_due > 0
* Send email notification
